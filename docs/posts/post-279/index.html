AIによる要約

概要と事実

Google Cloudは2026年6月13日（米国時間）、AIエージェントが組織内の知識を活用しやすくするためのナレッジ共有フォーマット「Open Knowledge Format（OKF）」を発表しました。初期バージョンであるOKF v0.1はGitHubで公開されており、マークダウンファイルとYAMLフロントマターを組み合わせた仕様です。必須フィールドはtypeのみで、title・description・resource・tags・timestampなどがオプションとして定義されています。例えば、BigQueryのテーブル情報を表すファイルでは、typeに「BigQuery Table」と指定し、タイトルや説明、リンク、タグ、タイムスタンプを付記します。こうしたファイルをディレクトリ構造で束ねた「OKFバンドル」は、マークダウンリンクで相互参照でき、グラフ構造の知識ベースとして機能します。一般的なマークダウンエディタで開けるため、人間も内容を理解しやすい形となっています。Google Cloudはリファレンス実装も同時に公開しており、特定のクラウドサービスやデータベース、モデルプロバイダー、エージェントフレームワークに依存しないオープン規格として提供しています。

ネットの反応と意見の変遷

発表直後、技術系フォーラムやSNSでは「オープンな知識共有フォーマットが登場した」と歓迎する声が多く見られました。特に、ベンダーロックインを避けたい企業や開発者からは、異なるAIエージェント間でのデータ連携が容易になると評価されています。一方で、実際に導入するには既存のドキュメントやツールをOKF形式に変換する手間が必要だという懸念も示され、移行コストやエコシステムの成熟度について議論が行われました。時間が経つにつれて、具体的なユースケースとしてナレッジベースの構築やAIエージェントのコンテキスト強化が挙げられ、コミュニティからはサンプルデータや変換スクリプトの共有を求める声が高まっています。全体としては、期待と実装へのハードルの両方が指摘されるバランスの取れた反応が続いています。

背景と結論

近年、大規模言語モデルやAIエージェントが情報を活用する際、単なるデータだけでなく文脈情報（コンテキスト）の質が結果の正確性に大きく影響することが明らかになっています。しかし、各ベンダーが独自のAPIやスキーマを提供しているため、組織内で異なるシステム間の知識を横断して活用する際に毎回実装を作り直す必要があり、これがAI活用のボトルネックとなっていました。OKFはこうした fragmentation（断片化）を解消するため、マークダウンという広く普及したフォーマットと軽量なYAMLメタデータを組み合わせ、誰でも容易に読み書きできるオープン仕様を提示しました。これにより、異なるツールやプラットフォーム間での知識の相互運用性が向上し、AIエージェントがより正確で実用的な回答を生み出す基盤が整うことが期待されます。結論として、OKFはオープンかつシンプルなナレッジ共有フォーマットとして、AIエージェントの実用化を後押しする可能性があり、今後のエコシステムの発展次第で業界標準となるかが注目されます。

グーグル、AIのための知識共有フォーマット「OKF」を公開

2026年6月15日 10:37

Google Cloudは13日(米国時間)、AIエージェントが組織内の知識などを活用しやすくするためのナレッジ共有フォーマット「Open Knowledge Format(OKF)」を発表した。初期バージョンの「OKF v0.1」としてGitHubで公開している。

LLMやAIエージェントが情報を活用する場合、与えられたデータの質とともに、文脈情報(コンテキスト)が重要になる。コンテキストが不足していると、LLMの本領が発揮できず、正確で実用的な結果を生み出すには、LLMに与える情報の内容とコンテキストの質が必要となる。

一方、AIエージェントが情報を横断して回答を組み立てるための文脈理解は、ベンダーごとに異なるAPIや、独自スキーマに対応する実装が毎回必要となっており、「同じ問題をゼロから解き直している」状態が、AI活用を進める多くの組織で発生しているという。こうした課題を解決し、ベンダーに依存しないオープン仕様として公開するものがOKFとなる。

OKFの核となるのは、MarkdownファイルとYAMLフロントマターの組み合わせだ。Markdownは、#で見出し、太字、- 箇条書きといった簡単な記号で文書を構造化できるテキスト記法で、GitHubやNotionなどの多くのツールで広く使われている。人間が直接読み書きできるうえ、プログラムからも解析しやすい点が特徴だ。

OKFではMarkdownに、YAMLと呼ばれる構造化データ記法でメタ情報を付加する。必須フィールドはtype(概念の種別)のみで、title・description・resource・tags・timestampなどがオプションとして定義される。

—type: BigQuery Table

title: Orders

description: One row per completed customer order.

resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders

tags: [sales, revenue]

timestamp: 2026-05-28T14:30:00Z

Schema

| Column | Type | Description |

|—————|———–|——————————————|

| order_id | STRING | Globally unique order identifier. |

| customer_id | STRING | FK to customers. |

Joins

Joined with customers on customer_id.

このようなファイルをディレクトリ構造で束ねたものが「OKFバンドル」で、ファイル間は通常のMarkdownリンクで相互参照できる。全体がグラフ構造の知識ベースとして機能する。OKFバンドルは一般的なMarkdownエディタで開けるため、人間も理解できる。

sales/├── index.md

├── datasets/

│ ├── index.md

│ └── orders_db.md

├── tables/

│ ├── index.md

│ ├── orders.md

│ └── customers.md

└── metrics/

│ ├── index.md

└── weekly_active_users.md

OKFでは最低限の規定だけを設けて、相互運用性を重視。特定のクラウドサービスや、データベース、モデルプロバイダー、エージェントフレームワークに縛られない、オープン規格として公開する。Google Cloudでは、リファレンス実装なども公開している。
